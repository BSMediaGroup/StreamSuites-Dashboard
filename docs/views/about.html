About StreamSuites

Current Phase

StreamSuites is advancing through a stabilization phase with a confirmed architectural win for Rumble chat via the platform’s canonical SSE stream. The dashboard remains a static surface that mirrors authoritative runtime exports without driving live automation, and now ships with a config state layer that hydrates from bundled JSON defaults or localStorage drafts and can export runtime-ready configuration for creators and platforms.

Scope for this repo: documentation-grade UI, local storage inspection, and read-only visibility into runtime-exported JSON snapshots.

Architecture & Separation of Concerns

  • StreamSuites Unified Runtime: Executes automation, enforces limits and quotas, connects to platforms, and remains the sole source of truth for chat triggers, jobs, and live state.
  • StreamSuites Dashboard (this repo): Static HTML/CSS/JS that runs in-browser, stores configuration locally, and visualizes runtime-exported state. No backend calls or live control paths are present.
  • Platform Clients: Per-platform adapters (Discord, Twitch, YouTube, Rumble, Twitter) live alongside the runtime and consume exported configuration. The dashboard reflects their expected contracts only.

Configuration, enforcement, observability, and execution are intentionally decoupled to preserve safety, auditability, and deterministic behavior.

Milestone: Rumble SSE Chat Ingestion

  • Rumble chat now uses the canonical SSE endpoint: https://web7.rumble.com/chat/api/chat/{CHAT_ID}/stream.
  • SSE stream emits init and messages events for reliable, scalable ingestion.
  • Runtime export alignment is in progress; dashboard hydration will follow once JSON snapshots are published.
  • Dashboard language stays neutral to polling vs SSE and renders runtime-exported state only.

No deprecated Rumble endpoints are assumed. The dashboard will remain read-only and rely on runtime-produced data drops.

Dashboard Capabilities (Static)

Overview

  • High-level storage status and configuration counts
  • Snapshot visibility for creators, triggers, jobs, limits, and platform telemetry when a runtime snapshot is present

Creators

  • Create and edit creator profiles
  • Platform enablement fields aligned with runtime schemas (YouTube, Twitch, Rumble, Discord)
  • Soft-disable/notes fields plus deterministic import/export as JSON
  • Hydrates from bundled defaults (`docs/data/creators.json`) or localStorage drafts

Chat Triggers

  • Define trigger phrases, match modes, responses, and cooldowns
  • Import and export trigger definitions

Jobs

  • Read-only statistics and breakdowns

Placeholder & Scaffolded Modules

  • Clips, Polls, Rate Limits, Settings, Chat Replay
  • Platform views for Rumble, YouTube, Twitch, Twitter, Discord (static visibility only)
  • Settings view now binds platform toggles and telemetry flags to platforms.json

Current Limitations & Deferred Features

  • No live platform connectivity from the dashboard; runtime exports remain authoritative.
  • No authentication, permissions, or role gating in this phase.
  • Chat Replay, Rate Limits, Clips, and Polls are UI placeholders pending runtime support.
  • Platform pages are read-only scaffolds; they do not initiate API calls or bot actions.
  • No backend APIs or live data bindings are assumed or invoked.

Dashboard Modes (Creator vs Admin)

The layout reserves space for future mode differentiation without enforcing it today:

  • Creator Mode: personal automation setup and visibility.
  • Admin Mode: cross-creator observability and system controls.

Mode-aware behavior is deferred until authentication and authorization are introduced.

About StreamSuites

Current Phase

StreamSuites is a modular livestream automation platform anchored by an enforcement-first runtime. The runtime owns all orchestration, scheduling, and quotas so behavior remains deterministic across Rumble, Twitch, YouTube, and Discord without platform-specific drift.

The dashboard and public surfaces are configuration- and visibility- driven. They render exported state from the runtime, provide creators a controlled way to author configs, and expose read-only public pages for clips, polls, and tallies without issuing live commands.

The platform is rolling out module by module. Runtime remains authoritative, the dashboard stays static by design, and public pages reflect shipped exports rather than live data pulls.

Architecture & Separation of Concerns

  • StreamSuites Unified Runtime: Executes automation, enforces limits and quotas, connects to platforms, and remains the sole source of truth for triggers, jobs, and livestream state.
  • StreamSuites Dashboard (this repo): Static HTML/CSS/JS that runs in-browser, stores configuration locally, and visualizes runtime-exported state. No backend calls or live control paths are present; public galleries for clips, polls, and tallies are read-only renders of runtime drops.
  • Platform Clients: Per-platform adapters (Discord, Twitch, YouTube, Rumble, Twitter) live alongside the runtime and consume exported configuration. The dashboard reflects their expected contracts only.

Configuration, enforcement, observability, and execution are intentionally decoupled to preserve safety, auditability, and deterministic behavior. Public surfaces remain read-only while runtime-side enforcement continues to expand.

Milestone: Rumble SSE Chat Ingestion

  • Rumble chat now uses the canonical SSE endpoint: https://web7.rumble.com/chat/api/chat/{CHAT_ID}/stream.
  • SSE stream emits init and messages events for reliable, scalable ingestion.
  • Runtime exports are published and feed the dashboard plus public chat-facing views.
  • Dashboard language stays neutral to polling vs SSE and renders runtime-exported state only.

No deprecated Rumble endpoints are assumed. The dashboard will remain read-only and rely on runtime-produced data drops.

Dashboard Capabilities (Static)

Overview

  • High-level storage status and configuration counts
  • Snapshot visibility for creators, triggers, jobs, limits, and platform telemetry when a runtime snapshot is present
  • Read-only public galleries for clips, polls, and tallies with detail pages driven by runtime exports

Creators

  • Create and edit creator profiles
  • Platform enablement fields aligned with runtime schemas (YouTube, Twitch, Rumble, Discord)
  • Soft-disable/notes fields plus deterministic import/export as JSON
  • Hydrates from bundled defaults (`docs/data/creators.json`) or localStorage drafts

Chat Triggers

  • Define trigger phrases, match modes, responses, and cooldowns
  • Import and export trigger definitions
  • Runtime-aligned exports back public surfaces without allowing live chat control from the dashboard

Jobs

  • Read-only statistics and breakdowns

Placeholder & Scaffolded Modules

  • Clips, Polls, Tallies, Rate Limits, Settings, Chat Replay
  • Platform views for Rumble, YouTube, Twitch, Twitter, Discord (static visibility only)
  • Settings view now binds platform toggles and telemetry flags to platforms.json

Current Limitations & Deferred Features

  • No live platform connectivity from the dashboard; runtime exports remain authoritative.
  • No authentication, permissions, or role gating in this phase.
  • Public pages for clips, polls, and tallies remain read-only; runtime mechanics continue to expand behind them.
  • Platform pages are read-only scaffolds; they do not initiate API calls or bot actions.
  • No backend APIs or live data bindings are assumed or invoked.

Dashboard Modes (Creator vs Admin)

The layout reserves space for future mode differentiation without enforcing it today:

  • Creator Mode: personal automation setup and visibility.
  • Admin Mode: cross-creator observability and system controls.

Mode-aware behavior is deferred until authentication and authorization are introduced.

Project Status

The dashboard tracks the StreamSuites runtime but does not replace it. Runtime contracts for jobs, triggers, quotas, clips, polls, and tallies are being exported incrementally so public pages can render without UI restructuring or live control paths.

The unified runtime is the automation engine that enforces quotas, schedules jobs, and coordinates platform adapters. Its goal is deterministic execution of the full StreamSuites feature set. Contracts and enforcement are locked and now emit the exports that drive public galleries; remaining effort focuses on deeper orchestration and rollout per module.

Trigger definitions run inside the runtime with deterministic matching, quotas, and cooldown enforcement. Exports now include trigger catalogs and response shapes that the dashboard and public pages render read-only. Next steps focus on broader platform parity and expanded moderation actions.

A cross-platform layer will normalize chat events across Rumble, Twitch, and YouTube so triggers, scoreboards, and moderation tools work uniformly. The desired outcome is consistent payloads regardless of source. Draft normalization rules exist, but runtime enforcement and exports are still early.

Dashboard ↔ Runtime Schema Contract
55% · Deterministic Exports + Public Gallery Alignment

The schema contract defines exactly how the dashboard consumes exported JSON for creators, platforms, public galleries, and telemetry. It now guarantees deterministic imports/exports for current modules and keeps expanding alongside new runtime drops.

Unified Live Chat Replay Module
35% · Runtime Signals + Export Shapes

Chat replay is meant to store and rehydrate message archives for post-stream review across platforms. The UI is staged to visualize those transcripts alongside metadata. Runtime signals exist, but stabilized export streams are still needed before artifacts can render.

WinForms GUI (Admin & System Controls)
20% · Planning

A WinForms control surface is planned for operators who prefer a desktop UI. It is intended to mirror admin views and telemetry without bypassing runtime safety. Planning and layout considerations are in place, but no runtime coupling or exports are implemented yet.

Clips Module
62% · Runtime Clipping + Public Gallery

The clips module automates livestream clipping, encodes segments with FFmpeg inside the runtime, and marches each clip through deterministic lifecycle states. Finished clips upload to the default StreamSuites Rumble channel today with exports that power dashboard and public gallery/detail pages. Current state: implemented in the runtime, visible across dashboard and public surfaces, with destination flexibility and moderation tooling still expanding.

Polls Module
58% · Runtime Polling + Public Gallery

Polls execute in the runtime with strict quotas and deterministic close conditions. Dashboard and public pages show read-only ballots, results, and detail pages sourced from runtime exports. Remaining work focuses on broader platform coverage, richer scheduling, and export cadence tuning.

Tallies Module
42% · Aggregation + Public Gallery

Tallies are programmatic, chat-driven aggregations (not direct votes) that bucket counts over weekly, monthly, or custom windows. Public gallery and detail pages are live and read-only, driven by runtime exports. Runtime-side mechanics for scheduled aggregation, triggers, and enforcement are underway.
